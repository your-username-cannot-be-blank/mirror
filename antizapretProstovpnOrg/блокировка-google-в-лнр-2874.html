drygdryg

Поиск Google на данный момент блокируют и по протоколу QUIC, и по HTTPS (TLS 1.2 и 1.3).
Чтобы обойти блокировку достаточно использовать zapret для Linux с обычной конфигурацией (которую можно получить посредством blockcheck.sh) и отключить QUIC в браузере (например, в Chrome/Chromium это делается на странице chrome://flags), потому что по умолчанию zapret не работает с QUIC. Либо можно включить обход QUIC в zapret, как показал это лично автор программы здесь с помощью флагов --dpi-desync-any-protocol --dpi-desync-cutoff=d4. Пример рабочего конфиг-файла (/opt/zapret/config):

Вы не отделаетесь одним конфигом. Если сделать только изменение в конфиге, вы ничего не достигните и сделаете только хуже. К тому же тот мой пост несколько устарел. Теперь nfqws может распознавать QUIC, может дешифровывать sni, поэтому any protocol больше не нужен.
Для обхода quic можно использовать custom script custom-nfqws-quic4all .
Но обязательно надо исследовать работает ли это вообще. Например, в основной России это уже практически не работает. Делает только хуже.
Если quic вызывает подвисания броузера, проще отключить его. Если же провайдер тупо блочит quic initial, то ничего не делать не надо. Броузер имеет алгоритм fast fallback на этот случай.
Проблемы начинаются, когда ответ на quic initial приходит, но дальше сессия блокируется в середине и подвисает. Именно это и наблюдается на русских провайдерах, если пробить quic initial

Общее впечатление - выборочная блокировка quic оказалась сложной задачей, потому этот протокол жестко режут. С разной степенью продвинутости на разных провайдерах
